HomeHub redesign — handoff spec

A self-contained brief for Claude Code: design system, screen inventory, and the rules to apply when extrapolating screens that aren't in the canvas. Read this before touching the Svelte code.

1 — Direction

The system is warm dark mode, anchored on an incandescent amber accent. It should feel like the room you're controlling — quiet, layered, lit from within — not like a generic dashboard. Surfaces are warm near-blacks; the "ON" state lights up with amber + soft glow, mirroring the lights themselves.

  • No emoji. Except in KidHome, which is the one place playful glyphs make sense.
  • No decorative SVG. Icons only where functional; placeholders + monospace explainers for imagery.
  • No gradients except for state shifts. The ON-state tile gradient and the day/night timeline gradient are the only allowed gradients.
  • Numbers in monospace. Every count, watt, temp, time uses Geist Mono with tabular figures. Labels stay in sans.

3 — Core primitives

Every screen composes from a small kit. Build these once in Svelte, then everything else falls out.

Tile

The most-used surface. Default is muted card; .tile.on shifts to a warm gradient with an amber border tint and the .tile-bulb indicator glows. This single transition does almost all visual storytelling.

/* default */
.tile { background: var(--card); border: 1px solid var(--hairline); border-radius: 22px; padding: 16px; }

/* on */
.tile.on {
  background: linear-gradient(155deg, #2b2419 0%, #221d14 60%, #1d180f 100%);
  border-color: rgba(245,189,110,0.18);
}
.tile.on .tile-bulb { background: var(--on); box-shadow: 0 0 24px 4px var(--on-glow); }

Switch

Two sizes: .sw (44×26, list rows) and .sw-big (64×36, hero). Spring easing on the thumb: cubic-bezier(0.34, 1.56, 0.64, 1). Off uses warm gray #b5b1a8, on uses pure white.

Chip

Pill, padding 7×12, font 12.5. Three states: default (muted card), .active (text-on-text), .on (amber soft + amber fg).

Rail

Slider track. .rail is 6px (inline). .rail-fat is 64px tall with embedded label + percent — used on Light detail for brightness.

Protocol badge

Uppercase 10px mono label + tiny line icon. Always uses the protocol accent. Never use the badge as a button.

4 — Layout rules

Mobile

  • Status bar reserved: top 54px is empty padding.
  • Bottom: 90px reserved for tab bar (60px bar + 30px safe area). All scroll content must use padding-bottom: 90px.
  • Tab bar items: 5 max. .tabbar > button.active tints amber.
  • Bottom sheets: 82% height by default, 28px top-radius, grabber + close X, sticky footer with primary (amber, 2fr) + optional secondary (card, 1fr).
  • Detail screens (Light detail, Matter steps): no tab bar — they get a back chevron in the chip-style.

Desktop (≥ 900px)

  • Fixed 240px nav rail, scrollable main. Same component for every page via DesktopShell.
  • Topbar: large h1 left, action buttons right. No breadcrumbs.
  • Device grid: 4 cols at 1280px. Drops to 3 at 1024, 2 at 768.
  • Notifications, add-device, and other transient surfaces open as popovers, not modals. See DesktopNotificationsPanel.

6 — Rules for extrapolating

Surfaces not in the canvas (group detail, scene detail, individual settings sub-screens, Tasmota commissioning) follow these rules:

  • List → detail. Detail screens get a back chevron in the chip-style top-left. Same 54px status pad. Tab bar hidden on detail.
  • Forms always live in a Sheet. Never push a full new screen for a form. Sheet height 82% by default; smaller (62–68%) for short forms like Timer.
  • Settings sub-screens use a card-grouped list pattern (see SettingsScreen) — never reach for a tabbed sub-nav.
  • Empty states use EmptyStatesScreen primitives: dashed border, dim icon, single CTA in .chip.on style. Mid-card, never full-screen.
  • Loading. Use the existing app's skeleton primitive (background shimmer on muted card) — don't add spinners.
  • Confirmations. Destructive actions get the ConfirmModalScreen pattern: centered card 340px max, danger-soft icon, Cancel left + danger-fill right.
Anti-patterns to avoid. No tabs inside views (use chip filters). No drawers from the left (use sheets from below). No purple/blue brand gradients (warm only). No pure-black surfaces. No emoji outside Kid mode. No icon-only buttons under 44×44 hit area.

7 — Motion

  • Press: scale to 0.97, 80ms. iOS-style squish, not translate.
  • Switch thumb: 220ms with spring cubic-bezier(0.34, 1.56, 0.64, 1).
  • Hover (desktop only): translateY(-1px) at 120ms ease. Disabled on touch via @media (hover: hover).
  • View transitions: 240ms fly-in (y: 10), 140ms fade-out. Already implemented in App.svelte.
  • Sheet open: 280ms cubic-out from the bottom. Backdrop fades in over 200ms.
  • Reduced motion: collapse all to 0.001ms (already in app.css).

8 — Notifications (NEW)

Two surfaces:

  1. Permission prompt — hero card with bell icon, explainer list of what gets pushed, big amber CTA. Shown on first sign-in (after dashboard renders) and from Settings.
  2. Inbox — bell icon in the home topbar opens the inbox (mobile: full-page; desktop: popover). Grouped by day, unread items have an amber dot + non-muted styling. Filter chips: All / Unread / Alerts / Schedules / Devices.

Alert tones map to icon color: info → cool, warn → #e8b96b, error → bad, success → good. The icon's background tile is --on-soft when unread, --card-3 when read.

9 — Spatial blueprint & Console (NEW)

Two specialised surfaces sitting outside the standard tile-pattern world:

  • Floor plan (SpatialBlueprintScreen) — the home as an architectural top-down SVG. Walls are 1.5px amber lines, door gaps are knocked out in bg color, furniture is faint subtle outlines. Each light is a positioned dot with a radial halo whose opacity tracks dim level. Renders on a 360×460 viewBox; ship as one SVG with light positions data-driven. Wears the same chrome as Light detail (back chip + centered title + Edit chip; Rooms tab active). Bottom strip lists active devices only — never the off ones.
  • Console (ConsoleScreen) — monospace power-user mode reached from Settings → System → Console. No tab bar. Devices render as a 5-column grid (dot · host · val · ASCII bar · age); the ASCII bar is literally █/░ characters in var(--font-mono). Live event tail uses ok/set/in/err tokens, colored from --good/--on/--cool/--bad. Background is #0a0907 (one shade deeper than --bg) to signal “raw mode.” Don’t round any corners or add shadows inside the console — that look is intentional.
Why two of them? Floor plan is the spatial way to read the home; Console is the textual way. Both are optional surfaces — most users never open Console — but they earn their keep with the self-hoster audience that runs HomeHub on a Pi.

10 — New user invite flow (NEW)

Endpoint: server emails a link with ?invite=TOKEN. App opens; LoginGate reads the param, calls api.lookupInvite, swaps the form for the set-password card.

  • Title: Welcome, {{username}} — pulled from the invite lookup.
  • Strength meter: 4 segments, all amber at 8+ chars with at least one letter + one digit.
  • On submit: api.acceptInvite(token, newPassword). Server sets session cookie + we drop the ?invite= param.
  • Invalid token: InviteInvalidScreen pattern (single card, danger-soft icon, "Go to sign in" link). Not in canvas — extrapolate from ConfirmModalScreen shape.

12 — Sanity checklist for Claude Code

  • Does every list row use the same .card + row pattern? (44–60px tall, 12–16px padding, icon left 36px wide, content middle, chevron/switch right)
  • Does every "ON" state use .tile.on's gradient + bulb glow? Don't substitute a simple background-color change.
  • Are all numbers in var(--font-mono) with "tnum" 1 feature setting?
  • Are all section headers 17px / 600 / left-padded 22px (mobile) or 0 (desktop)?
  • Did you remove emoji from non-Kid surfaces?
  • Is the tab bar hidden on every detail/form/Matter step screen?
  • Is the notification dot exactly 7×7 amber, not the old 8×8 indigo?

Questions during build? The canvas at index.html is the source of truth — open it side-by-side. Open any artboard fullscreen via the Focus icon (top-right of each tile) for pixel measurements.
